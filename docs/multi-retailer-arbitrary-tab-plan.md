# Plan: Arbitrary-tab tooling (Refract-style) + multi-retailer (Walmart)

**Goal:** Evolve Target Checkout Helper into a **retail checkout companion** that can (a) attach to **whichever tab** the user is driving, (b) use **CDP / cookies / optional DNR** where justified, and (c) add **Walmart** (and later others) without rewriting the core each time.

**Principles:** Same-repo **modular monolith** first (no build step until pain forces it); **host allowlist in code** even if manifest is broad; **retailer = plug-in surface** (selectors + flow + optional API); **two-trust tiers**: “store / public” manifest vs “full / sideload” manifest.

---

## 1. Target architecture

### 1.1 Two manifest profiles (same JS, different `manifest.json`)

| Profile | `host_permissions` | `debugger` | Intended use |
|---------|------------------|------------|----------------|
| **Retail-full** | `<all_urls>` (or explicit `*://*.target.com/*`, `*://*.walmart.com/*`, …) | yes | You + power users, sideload or private listing |
| **Store-safe** | Only origins you can justify in listing | no (or optional) | Chrome Web Store, minimal prompts |

Ship **two folders** or one `manifest.store.json` / `manifest.full.json` copied to `manifest.json` in a tiny script—**no bundler required** if you keep vanilla JS.

### 1.2 Core vs retailer modules

```
target-checkout-helper/
  manifest.json              # profile-specific
  background.js              # thin router: messages, alarms, attach policy
  core/
    hosts.js                 # ALLOWED_RETAILERS, matchUrlToRetailer(), normalize
    debuggerBridge.js        # attach/detach/sendCommand; tab allowlist
    cookiePool.js            # generic harvest (retailer tag + domain filters)
    messaging.js             # shared message types
  retailers/
    target/
      manifest-snippet.json  # optional: content_scripts matches for merge tooling
      content.js             # moved from root or re-export shim
      selectors.js           # SEL, page types, checkout steps
      fulfillment.js         # RedSky / ATC logic
    walmart/
      content.js
      selectors.js
      fulfillment.js         # if any public API; else DOM-only
  popup/ ...                 # or keep flat popup.* with retailer picker
```

**Loader pattern:** `background.js` does `importScripts('core/hosts.js', 'core/debuggerBridge.js', …)` and **one** `importScripts` per active retailer build, **or** single `retailers-bundle.js` generated by a one-line Node script later.

**Content scripts in manifest:** Use multiple `content_scripts` entries:

```json
"content_scripts": [
  { "matches": ["*://*.target.com/*"], "js": ["dropPollingTiming.js", "retailers/target/bootstrap.js"], "run_at": "document_end" },
  { "matches": ["*://*.walmart.com/*"], "js": ["dropPollingTiming.js", "retailers/walmart/bootstrap.js"], "run_at": "document_end" }
]
```

Each `bootstrap.js` is tiny: set `window.__TCH_RETAILER = 'target'|'walmart'` then `importScripts` is not available in content—**content must be single file or concatenated**. Practical vanilla approach: **one `content.js`** that starts with `const RETAILER = detectRetailer(location.hostname)` and switches to `targetHandlers` / `walmartHandlers` objects in same file or `import()` only if you add ES modules + type="module" (MV3 supports module content scripts). Simplest path for “no build”: **single content entry** with internal `if (hostname)` branches and split logic across `retailers/target/*.js` merged manually or via a 10-line concat script.

**Recommendation for v1 multi-retailer:** One `content.js` with `switch (retailer) { case 'target': … case 'walmart': … }` and **separate files** included in manifest **array** `["retailers/shared.js", "retailers/target.js", "retailers/walmart.js", "content-entry.js"]` in order—Chrome loads in sequence; shared + per-retailer + thin entry.

## 2. Arbitrary-tab / Refract-style capabilities

### 2.1 Capabilities map

| Capability | API | Use |
|------------|-----|-----|
| Any-tab cookie harvest | `cookies` + `<all_urls>` | Snapshot current tab’s first-party cookies for **matched retailer host** only (enforce in code). |
| Attach debugger | `chrome.debugger` | Network tap, harder SPA steps, optional `Fetch.continueRequest` (advanced). |
| Tab targeting | `tabs`, `tabCapture` (optional) | “Use active tab”, “use tab id from monitor”. |
| Optional DNR | `declarativeNetRequest` + dynamic rules | Block ad iframes / noise per retailer (like Refract’s `blocking.json`)—**separate permission**; add only if measured win. |
| Native messaging (optional later) | `nativeMessaging` | Desktop “brain” like Refract—**out of scope** for first milestone unless you need it. |

### 2.2 Security model (non-negotiable)

1. **`matchRetailer(url)`** — If URL is not Target/Walmart (config list), **no** cookie export, **no** debugger attach, **no** automation messages—return `unknown`.
2. **User gesture** for debugger attach and for “harvest this tab” when tab is not allowlisted retailer (if you ever allow generic tab, gate behind **explicit** toggle + warning).
3. **Session storage pools** keyed by `retailerId`**: `tchHarvest:target`, `tchHarvest:walmart` or single array with `{ retailer, … }`.
4. **Logging:** Never log cookie values or auth headers.

### 2.3 Debugger bridge (concrete flow)

1. Popup: “**Attach to active tab**” → `chrome.tabs.query({ active: true, currentWindow: true })` → validate URL via `hosts.js` → `chrome.debugger.attach({ tabId }, '1.3', onAttach)`.
2. Enable domains: `Network.enable`, optionally `Fetch.enable` with `patterns` for document + XHR.
3. Forward interesting events to SW state or post to content via `chrome.tabs.sendMessage` (content may not see CDP; **SW receives events**).
4. `detach` on timeout, on tab close (`tabs.onRemoved`), on user “Stop”, on `debugger.onDetach`.

**Walmart note:** DOM and network names differ; CDP is useful earlier for **discovery** (record HAR file, map checkout XHR) before hardcoding selectors.

## 3. Walmart extension (practical discovery order)

Walmart checkout is a **different SPA**, different classes, likely **no RedSky**. Plan:

| Phase | Work |
|-------|------|
| **W0 — Recon** | Manual checkout recording: DevTools **Performance + Network**, export HAR; note cart, checkout, sign-in URLs (`walmart.com`, `grocery.walmart.com` may differ—**pick one scope first**, e.g. `www.walmart.com` only). |
| **W1 — Page model** | Implement `getPageType()` analog: `pdp`, `cart`, `checkout`, `signin`, `unknown`. |
| **W2 — Happy path** | PDP → ATC → cart → checkout → shipping/payment/review (mirror TCH policy: **no place order** unless explicit dangerous flag). |
| **W3 — Monitor** | If Walmart exposes inventory JSON from PDP, mirror `background.js` poll pattern with **Walmart-specific** URL + parser; else DOM/streaming fallback like Target HTML path. |
| **W4 — Harvest** | Same cookie pool pipeline; domain filter `walmart.com` + `www.walmart.com`. |

**Manifest hosts:** At minimum `*://*.walmart.com/*` and whatever host actually serves checkout if different (verify in HAR).

## 6. Risks and mitigations

| Risk | Mitigation |
|------|------------|
| Store rejection for `<all_urls>` | Store-safe manifest; broad hosts sideload-only or optional permissions. |
| Accidental cross-site cookie access | `hosts.js` central gate; unit-style manual checklist per release. |
| Debugger left attached | Timeouts + `tabs.onRemoved` + popup “Stop”. |
| Walmart DOM churn | Selector indirection table + versioned `selectors.js`; CDP logging in dev only. |

**Bottom line:** Treat **“arbitrary tab”** as **“user-selected tab that passes `hosts.js`”**, not “any origin read.” Treat **Walmart** as a **second retailer object** behind the same orchestration. Ship **broad permissions only in the full profile**, keep the **store profile** honest and narrow, and **refactor Target out of a monolithic `content.js` first** so Walmart is additive work, not a fork.
